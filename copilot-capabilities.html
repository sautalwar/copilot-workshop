What is GitHub Copilot?

Your AI pair programmer — three tools in one

Code Completion

💬

Chat Assistant

Ask questions, get explanations, debug issues interactively

Chat Modes

Three gears of interaction

🔍

Ask Mode

Read-only exploration.
Answers questions, explains code.
Never modifies files.

✏️

Edit Mode

Suggests changes inline.
You review & accept each diff.
Collaborative editing.

🤖

Agent Mode

Full autonomy.
Runs commands, edits files,
iterates until done.

@ Commands

Scope the conversation to the right context

  • @workspace — Search & reason across your entire codebase
  • @vscode — VS Code settings, keybindings & extension APIs
  • @terminal — Explain errors & suggest shell commands
  • @github — Pull requests, issues, repos & Actions workflows
  • @azure — Azure services, deployment & cloud resources

/ Commands

Quick actions — no prompt engineering needed

  • /explain — Break down selected code in plain English
  • /fix — Propose a fix for problems in selected code
  • /new — Scaffold a new workspace from a description
  • /newNotebook — Create a Jupyter notebook from a topic
  • /compact — Summarize the conversation to save context

# Commands

  • #file:app.ts — Attach a specific file for reference
  • #selection — Include the currently selected code
  • #terminalLastCommand — Attach the last terminal output
  • #codebase — Let Copilot search the full project for context

📱 "Giving Copilot a Phone"

Without MCP

Copilot is incredibly smart but can only see your open files. It can't check your database, call your API, or read another repo.

"I'd love to help, but I can't see your production data..."

Copilot can reach out to databases, APIs, and services — with safety guardrails at every step.

Demo 18: Building an MCP Server from Scratch

It's surprisingly simple — import SDK, create server, register tools. The magic is the redaction layer.

"Building an MCP server is surprisingly simple — import the SDK, create a server, register your tools. The real magic isn't the server itself, it's the redaction layer you put between the data and the LLM."

📦 Step 1: Import the SDK

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";

⚙️ Step 2: Create the Server

const server = new McpServer({
  name: "citizen-data-server",
  version: "1.0.0"
});

🔧 Step 3: Register a Tool

server.tool(
  "list-citizens",
  "List all citizens with PII redacted",
  { limit: z.number().optional() },
  async ({ limit }) => {
    const citizens = await getAllCitizens(limit);
    const safe = citizens.map(redactCitizen); // ← THE KEY LINE
    return { content: [{ type: "text", text: JSON.stringify(safe) }] };
  }
);

🛡️ Step 4: The Redaction Functions

// services/redaction-service.js

function maskSSN(ssn) {
  return ssn.replace(/\d{3}-\d{2}/, "***-**");  // 123-45-6789 → ***-**-6789
}

function maskEmail(email) {
  const [user, domain] = email.split("@");
  return `${user[0]}***@${domain}`;       // john@gov → j***@gov
}

function maskPhone(phone) {
  return phone.replace(/\d(?=\d{4})/g, "*"); // 555-123-4567 → ***-***-4567
}

📋 Step 5: Audit Logging

// security/audit-logger.js

function logAccess(toolName, params, recordCount) {
  console.log(JSON.stringify({
    timestamp: new Date().toISOString(),
    tool: toolName,
    params: params,
    records: recordCount,
    redacted: true
  }));
}

Demo 19: Safe vs Unsafe — The 3-Line Difference

The difference between leaking SSNs and being compliant is ONE line of code.

"The difference between leaking Social Security Numbers and being fully compliant is ONE LINE — .map(redactCitizen). Let me show you exactly what that looks like."

🔴 vs 🟢 Side by Side

🚨 UNSAFE — Raw PII Exposure

// ❌ DON'T DO THIS
const citizens = await getAllCitizens();

return {
  content: [{
    type: "text",
    text: JSON.stringify(citizens) // ← RAW DATA!
  }]
};

🚨 What Copilot sees:

{ "name": "John Smith", "ssn": "123-45-6789", "email": "john@agency.gov", "phone": "555-123-4567" }

✅ SAFE — PII Redacted

// ✅ DO THIS
const citizens = await getAllCitizens();
const safe = citizens.map(redactCitizen); // ← ONE LINE
return {
  content: [{
    type: "text",
    text: JSON.stringify(safe) // ← CLEAN DATA
  }]
};

✅ What Copilot sees:

{ "name": "John Smith", "ssn": "***-**-6789", "email": "j***@agency.gov", "phone": "***-***-4567" }

📐 The Exact Difference

JSON.stringify(citizens) → citizens.map(redactCitizen)

One line. That's the entire difference between a data breach and compliance.

🛡️ What redactCitizen Does

function redactCitizen(citizen) {
  return {
    ...citizen,
    ssn:   maskSSN(citizen.ssn),     // 123-45-6789 → ***-**-6789
    email: maskEmail(citizen.email), // john@gov   → j***@gov
    phone: maskPhone(citizen.phone), // 555-1234   → ***-1234
  };
}

Demo 20: Configuring MCP in VS Code

One JSON file tells VS Code where to find your MCP servers and how to connect.

"This JSON file is all it takes — it tells VS Code where to find your MCP servers, how to launch them, and what environment variables to pass. Drop it in your repo and every developer gets the same tools."

📄 .vscode/mcp.json

📁 .vscode/mcp.json 
// .vscode/mcp.json — MCP Server Configuration
{
  "servers": {
    "citizen-data": {
      "command": "node",
      "args": ["./mcp-server/index.js"],
      "env": {
        "DB_PATH": "${workspaceFolder}/data/citizens.db",
        "AUDIT_LOG": "${workspaceFolder}/logs/audit.log",
        "REDACTION_ENABLED": "true"
      }
    }
  }
}

🔍 Breaking It Down

📛 Server Name

"citizen-data"

A unique identifier for your server. Copilot shows this when listing available tools.

⚡ Command

"node"

What to run. Can be node, python, npx, docker — anything that starts your server.

📝 Args

["./mcp-server/index.js"]

Arguments passed to the command. Points to your server's entry file.

🔐 Env Variables

DB_PATH, AUDIT_LOG, ...

Environment variables injected at launch. Supports ${workspaceFolder} for portable paths. Never hardcode secrets here — use .env files.

💡 Pro Tips

1
Commit this file to your repo — every developer gets the same MCP tools automatically
2
Use ${workspaceFolder} — makes paths work on any machine
3
Keep secrets in .env — reference via ${env:SECRET_NAME}, never inline
4
Multiple servers are OK — add more entries under "servers" for different data sources

on:                            ◀ When does it run?
  push:

  pull_request:

🧠 Copilot Memory — Three Levels

How Copilot remembers context across different scopes

🧠

User Memory

SCOPE: You, across all repos
  • Your coding preferences
  • Preferred languages & frameworks
  • Personal style choices
  • Persists across sessions
💡 "I always prefer async/await over .then()"
📁

Repository Memory

SCOPE: Team rules in the repo
  • copilot-instructions.md
  • Prompt files & custom agents
  • Shared with entire team
  • Version controlled in Git
💡 "All APIs must use pagination"
💬

Session Memory

SCOPE: Within one chat
  • Current conversation context
  • Files you've referenced
  • Decisions made this session
  • Resets when chat closes
💡 "We decided to use PostgreSQL"
⬆️ Priority: Session Memory overrides → Repository Memory overrides → User Memory (most specific wins)

👁️ What Copilot CAN vs CANNOT See

Copilot isn't spyware — only sees what you explicitly share

✅ CAN SEE
✅
Open files in editor
Currently open tabs provide context
✅
#file references
Files you explicitly attach with #
✅
@workspace indexed files
Workspace-wide search when you use @workspace
✅
MCP tool outputs
Data returned from connected MCP servers
✅
Instruction files
copilot-instructions.md and custom instructions
❌ CANNOT SEE
❌
Other files on disk
Unopened files outside the workspace
❌
Other repositories
Can't see code in repos you don't have open
❌
Browsing history
No access to your browser or web activity
❌
Email / Slack / Teams
No access to communication tools
❌
Closed chat sessions
Previous conversations are not accessible
🔑 Rule of thumb: If you didn't open it, reference it, or connect it via MCP — Copilot doesn't know it exists.
