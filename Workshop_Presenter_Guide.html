📚 DEEP DIVE: Instructions vs Prompts vs Agents vs Skills

When to Use Which — A Decision Framework

🧭 The Confusion

Common question: "Aren't prompts and instructions the same thing?"

Answer: NO. They're fundamentally different tools for different purposes.

🔍 The Four Mechanisms

Mechanism What It Is When To Use Example
Instructions Persistent context that applies to ALL Copilot interactions in the repo Coding standards, architecture rules, naming conventions that never change "Use constructor injection. Never use @Autowired on fields."
Prompts Reusable templates for SPECIFIC tasks (file-scoped or repo-scoped) Repetitive workflows: "Write tests for this class" or "Fix this bug" A .prompt.md file with: "Analyze and fix the bug. Add regression test."
Agents Chat mode personas ("Be X, do Y") Role-playing for specialized tasks: "You are a security auditor" "Be a code reviewer. Review this PR for security issues only."
Skills Executable workflows with tools (CLI-only, advanced) Complex multi-step automation: deploy to Azure, run compliance checks azure-prepare skill that generates infra, Dockerfiles, and deploys

🎯 Decision Flowchart (Text-Based)

📜
If the rule applies to EVERY piece of code in the repo...
→ Use Instructions (.github/copilot-instructions.md)
📋
If you do the SAME task repeatedly (write tests, fix bugs, add endpoints)...
→ Use Prompts (.github/prompts/*.prompt.md)
🤖
If you need a SPECIALIZED perspective for one conversation...
→ Use Agents (Chat mode with persona: "Be X")
⚙️
If you need MULTI-STEP automation with external tools...
→ Use Skills (CLI workflows, advanced)

🎭 Real-World Scenarios

Scenario: "I want Copilot to always use SLF4J logging, never System.out.println."
Solution: Instructions. Add to .github/copilot-instructions.md: "SLF4J for logging — never System.out.println"
Scenario: "I need to write tests for 20 classes this sprint."
Solution: Prompt. Create .github/prompts/write-tests.prompt.md with a template: "Analyze the source, identify all code paths, write JUnit 5 tests with @WebMvcTest..."
Scenario: "I want Copilot to review this PR like a security expert."
Solution: Agent. In Chat, say: "Be a security auditor. Review this PR for SQL injection, XSS, secrets exposure. Ignore style issues."
Scenario: "I need to deploy this app to Azure with all infrastructure."
Solution: Skill (CLI). Use azure-prepare skill to generate Bicep/Terraform, Dockerfiles, and azd configs.

⚠️ Common Misconception

WRONG: "I'll put my 'write tests' template in copilot-instructions.md so it's always available."

WHY IT'S WRONG: Instructions apply to EVERY interaction. You'll pollute your context with task-specific instructions that don't apply when you're just writing a new method.

RIGHT: Put task templates in Prompts. Invoke them ONLY when you need them.

☕ DEEP DIVE: Java Development with Agentic AI

Gotchas, Best Practices, and Sample Prompts

✅ What Agent Mode Handles Well for Java/Spring Boot

  • Adding a full Controller + Service + Repository endpoint: "Add a DELETE /api/orders/{id} endpoint with cascade delete logic and tests"
  • Writing JUnit 5 tests: "Write integration tests for OrderController using @WebMvcTest and MockMvc"
  • Refactoring with safety: "Refactor OrderService to use Optional instead of returning null"
  • Following Spring conventions: Constructor injection, @Transactional, ResponseEntity with correct status codes

⚠️ Common Spring Boot Gotchas

Gotcha: Lazy loading exceptions (accessing @OneToMany outside transaction).
Fix: Add @Transactional(readOnly = true) to service methods. Agent Mode sometimes forgets this.
Gotcha: Missing @Valid on @RequestBody → validation doesn't trigger.
Fix: In custom instructions, add: "Use @Valid on all @RequestBody parameters"
Gotcha: Wrong HTTP status codes (200 for creates instead of 201).
Fix: In custom instructions: "POST returns 201 Created with Location header"
Gotcha: Skipping the service layer (controller calls repository directly).
Fix: In custom instructions: "Never skip the service layer — controllers must not call repositories directly"

🎯 Sample Prompts That Work Great

// Adding an endpoint
"Add a GET /api/inventory/low-stock endpoint that returns all items
with quantity < 10. Include service logic and integration tests."

// Writing tests
"Write comprehensive JUnit 5 tests for InventoryService. Cover happy path,
edge cases (empty results, invalid SKU), and exception handling."

// Debugging
"The order confirmation fails when inventory is low. Debug OrderService.updateOrderStatus()
and fix the issue. Add a test that reproduces the bug."

// Refactoring
"Refactor OrderController to use Java records for request/response DTOs instead of
plain classes. Update all endpoints and tests."

🚀 When Agent Mode Helps vs Manual Coding

Task Agent Mode Manual Coding
Adding boilerplate CRUD endpoints ✅ Faster — generates controller, service, tests in one go ❌ Tedious — lots of repetitive typing
Complex business logic with edge cases ⚠️ Needs review — AI might miss subtle requirements ✅ You understand the domain better
Writing 50+ unit tests ✅ Agent Mode excels — follows your test template ❌ Soul-crushing repetition
Debugging a production issue ⚠️ Helpful but not autonomous — use Chat mode ✅ You know the system state

🔌 DEEP DIVE: MCP Architecture Explained Simply

📖 The USB-C Analogy (Expanded)

Before MCP: Every AI tool had custom integrations. Copilot had a Jira plugin, Claude had a different Jira plugin, ChatGPT had yet another. Like every device needing a different cable.

After MCP: One standard protocol. You build ONE Jira MCP server, and ANY MCP-compatible AI can use it. Like USB-C — one cable, all devices.

🎤 Explaining MCP to a Non-Technical Manager (30 seconds)

"MCP is like a translator. It lets Copilot talk to your internal tools — Jira, databases, wikis — without hardcoding each integration. You build one 'translator' server, and Copilot can read your data securely. It's a standard protocol, like HTTPS, so it works with any AI tool, not just Copilot."
