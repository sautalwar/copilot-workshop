The Problem: You're Paying Twice

Every CI/CD run that touches AWS data hits you with two bills.

🚨 AWS Data Egress

Every byte leaving AWS costs $0.09/GB. An 8 GB dataset download per run = $0.72 per run.

  • S3 → Internet egress charges apply to every CI job
  • Multiply by 50 workflows × 20 runs/month
  • No caching between ephemeral runners

💰 GitHub Runner Minutes

GitHub-hosted runners bill at $0.008/min (Linux). A 15-minute job = $0.12 per run.

  • Data-heavy jobs spend most time downloading, not building
  • You pay for wait time, not just compute time
  • Larger runners cost 2–10× more per minute
$0.72 egress  +  $0.12 runner  =  $0.84 / run  ×  50 workflows  ×  20 runs/mo  =  $840 / month  ≈  $10,080 / year
💡 CONCEPT: AWS Data Egress
What it is: Every time data leaves AWS (goes to the internet or another cloud), AWS charges you a fee. It's like a toll booth — you pay to take your data out.
Why it matters: If your CI/CD pipeline downloads 8 GB of data from S3 every run, you pay ~$0.72 each time. Do that 20 times a month across 50 workflows, and it adds up fast.
Use cases: Any workflow that needs to process data stored in AWS — ML training, ETL pipelines, integration testing with production data.
Honest gap: AWS CodeBuild running IN the same AWS region has zero egress cost. We should acknowledge this. Our advantage: you stay in GitHub + get persistent caching.
AI opportunity: An AI agent could analyze your AWS billing data and automatically identify which workflows generate the most egress cost, prioritizing migration.

Why It Hurts: Ephemeral = No Memory

GitHub-hosted runners are brand-new every run. Nothing survives.

💡 AHA: The data hasn't changed between runs — but because each runner starts from zero, you re-download and re-pay every time. Persistent storage eliminates this entirely.
💡 CONCEPT: Ephemeral Runners
What it is: A runner that is created fresh for each job and destroyed after. Like a hotel room that gets deep-cleaned between every guest.
Why it matters: Great for security (no leftover secrets), bad for data (no persistent storage). Your 8 GB dataset doesn't survive between runs.
Tradeoff: Ephemeral IS the right choice for security-sensitive jobs. We're not saying ephemeral is bad — it just doesn't fit data-heavy workloads.
Honest gap: GitHub Actions caching (actions/cache) helps for dependencies, but has a 10 GB repo limit and doesn't cover large datasets well.
WHAT YOU'RE SHOWING: A visual diagram of three consecutive workflow runs, each downloading the same 8GB dataset from scratch. The "destroyed" labels and red X marks make the waste visceral. WHY: This slide answers the "why can't you just cache it?" question before it's asked. The audience sees that ephemeral runners fundamentally cannot hold data between runs. WHAT TO SAY: "Here's what's happening behind the scenes. Run 1 spins up a brand-new VM, downloads 8 gigs from S3, runs your build, then — poof — the whole machine is destroyed. Run 2 starts a completely different VM. It has no idea Run 1 ever happened. So it downloads the same 8 gigs again. And again for Run 3. Same data. Copied three times. Paid three times." Solution: AKS Runner Fleet with Persistent Cache

💾 Azure Files PVC

Three innovations working together:

1. Actions Runner Controller

ARC auto-scales runner pods on demand. Zero pods when idle, instant scale-up on workflow_job webhook.

2. Persistent Volume Claims

Azure Files PVC survives pod restarts. Download 8 GB once — every future run mounts the same cache in seconds.

3. Smart Routing

Workflow labels (runs-on: [self-hosted, aks]) route data-heavy jobs to AKS, while quick jobs stay on GitHub-hosted.

💡 CONCEPT: Actions Runner Controller (ARC)
What it is: A program that runs inside your Kubernetes cluster and automatically creates/destroys GitHub Actions runner pods based on demand. Think of it as a smart manager that hires temporary workers when there's work and sends them home when it's done.
Why it matters: Without ARC, you'd have to manually start/stop runner machines. ARC does it automatically, responding to GitHub webhooks in real time.
Honest gap: ARC adds Kubernetes complexity. If the customer has no K8s experience, there's a learning curve. Jenkins agents are simpler to set up (but harder to scale).
AI opportunity: Copilot can generate the ARC Helm configuration for you based on your workflow requirements.
💡 CONCEPT: Persistent Volume Claim (PVC)
What it is: A request for storage in Kubernetes. Think of it like renting a storage locker — your data stays there even after the runner pod is destroyed. Next time a runner starts, it opens the same locker and finds the data still there.
Why it matters: This is THE key innovation. Without PVC, you'd have to re-download 8 GB every run (just like GitHub-hosted). With PVC, you download once and cache forever.
Cost: PVC adds storage cost (~$0.06/GB/month). For 10 GB, that's $0.60/month. Trivial compared to the egress savings.
WHAT YOU'RE SHOWING: The architecture slide — the "how we fix it" answer. A flow diagram from GitHub Actions through AKS to persistent storage, plus three innovation cards. WHY: After two slides of pain, the audience is ready for the solution. This slide needs to feel simple despite the underlying complexity. The flow boxes make it linear and approachable. WHAT TO SAY: "Here's the architecture. Your workflow triggers in GitHub exactly like it does today. But instead of spinning up an ephemeral VM, the job is routed to an AKS cluster where Actions Runner Controller — ARC — automatically creates a runner pod. That pod mounts a Persistent Volume Claim — a shared disk — that has your dataset already cached from the last run. No download. No egress. Build starts immediately." KEY EMPHASIS: "Nothing changes in your workflow YAML except the runs-on label. Your developers don't need to learn Kubernetes." Runner Types at a Glance

🏠 On-Premises

  • Full hardware control
  • Zero egress (data stays local)
  • GPU / specialized hardware
  • Compliance-friendly (data sovereignty)
  • You manage everything
  • No auto-scaling
  • Always-on cost

☷ AKS Self-Hosted

  • Persistent cache (PVC)
  • Scale-to-zero (pay only when building)
  • Custom images & toolchains
  • Azure-managed Kubernetes
  • ARC handles auto-scaling
  • ~30s cold-start from zero
  • Requires K8s familiarity for setup

☁ GitHub-Hosted

  • Zero setup / maintenance
  • ~5s startup time
  • Pre-installed tool ecosystem
  • Pay-per-minute pricing
  • Ephemeral (no persistent storage)
  • Egress costs from other clouds
  • Limited customization
🎯 The goal is NOT to replace GitHub-hosted runners — it's to use the right runner for the right job. Quick lints stay on GitHub-hosted; data-heavy builds go to AKS.
💡 CONCEPT: Runner Labels / runs-on
What it is: Labels are like name tags you put on runners. In your workflow YAML, you say runs-on: [self-hosted, aks, gpu] and GitHub matches your job to a runner with ALL those labels. Like ordering food — "I want something hot, vegetarian, and spicy" — only restaurants matching all three criteria qualify.
Why it matters: This is how you route different jobs to different runner types without changing your workflow logic.
AI opportunity: An AI agent could analyze your workflow's resource usage and automatically suggest optimal labels.

💡 CONCEPT: Scale-to-Zero
What it is: When no CI jobs are running, the AKS runner pods are completely shut down. Zero pods = zero compute cost. When a new job arrives, ARC automatically spins up a runner in ~30 seconds. Like a light that turns off when you leave the room.
Why it matters: You ONLY pay for compute when builds are actually running. GitHub-hosted does this too (you only pay per-minute), but with AKS you get persistent storage AND scale-to-zero.
Honest gap: There's a ~30 second cold-start delay when scaling from zero. GitHub-hosted runners start in ~5–10 seconds. For most workflows this doesn't matter, but mention it honestly.
