💡 Q&A – Technical Depth

Top questions from senior engineers – with prepared answers

Q: How does delta sync work for S3-based caches?
A: aws s3 sync uses ETags (MD5 checksums) and last-modified timestamps to compare local vs. remote objects. Only changed files are transferred – identical files are skipped entirely. For a 2 GB node_modules cache where 50 MB changed, only that 50 MB moves. Think of it like rsync but for object storage. The ETag comparison happens server-side, so you don’t even download file headers for unchanged objects.
Q: How does RDS change detection work without polling?
A: We use RDS Event Subscriptions → SNS → Lambda. RDS emits events for snapshot completion, failover, parameter changes, etc. The SNS topic fans out to a Lambda that triggers the appropriate GitHub Actions workflow via repository_dispatch. Latency is typically under 30 seconds. No polling, no cron – pure event-driven.
Q: What happens if PVC storage fills up?
A: We set alerts at 80% capacity via Prometheus kubelet_volume_stats_used_bytes. If it fills, the runner pod gets an I/O error and the job fails gracefully – it doesn’t crash the node. Recovery: expand the PVC (if using a CSI driver that supports expansion), or run a cache-cleanup job that prunes entries older than N days. AKS supports online PVC expansion for Azure Disk and Azure Files.
Q: Can multiple workflows share the same cache volume?
A: Yes, with caveats. Use ReadWriteMany (RWX) access mode with Azure Files. Multiple pods can mount the same PVC simultaneously. For ReadWriteOnce (RWO) volumes like Azure Disk, only one pod can mount at a time – use separate PVCs per runner or schedule sequentially. We recommend RWX for shared caches and RWO for runner-specific scratch space.
Q: What happens if the AKS cluster itself fails?
A: The workflow YAML uses a runs-on label fallback strategy. If the self-hosted runner is unavailable for >5 minutes, the job can be configured to fall back to ubuntu-latest using a matrix strategy or a reusable workflow with conditional runner selection. The AKS control plane has a 99.95% SLA (with uptime SLA enabled). For node failures, the cluster autoscaler replaces nodes within 2–4 minutes.
Q: How do we manage custom runner images?
A: Build custom images using a Dockerfile based on ghcr.io/actions/actions-runner. Store in ACR (Azure Container Registry) with geo-replication. Use a GitHub Actions workflow to build and push the image on changes – dogfooding at its best. ARC references the image in the RunnerDeployment spec. Tag images with semver and pin runner deployments to specific versions for stability.

🔒 Q&A – Security & Compliance

Questions from CISOs and security architects

Q: Where does our data physically reside during CI/CD?
A: With self-hosted runners on AKS, all build artifacts, source code, and secrets stay within YOUR Azure tenant and subscription. Code is cloned into the runner pod, processed there, and artifacts are pushed to YOUR storage. GitHub’s control plane only sees job metadata: start time, duration, status, labels. No source code or build output ever transits GitHub’s infrastructure.
Q: Can this architecture meet HIPAA / SOC2 requirements?
A: Yes. AKS runs in your HIPAA-eligible Azure subscription. Use dedicated node pools with encryption at rest (Azure Disk Encryption) and in transit (TLS 1.2+). Enable Azure Policy for compliance guardrails. For SOC2, the audit trail comes from GitHub’s audit log API + Azure Monitor + Defender for Containers. The key: data never leaves your compliance boundary.
Q: Does GitHub see our source code or secrets?
A: No. With self-hosted runners, GitHub sends a job payload (workflow YAML instructions) to the runner. The runner pulls code directly from your repo using a short-lived token. Secrets are injected from GitHub’s encrypted vault into the runner environment – they’re never logged or persisted. GitHub never has access to your runner’s filesystem, network, or compute resources.
Q: How is network isolation achieved?
A: AKS runners operate inside your VNet. Use Network Policies (Calico or Azure NPM) to restrict pod-to-pod traffic. Place runners in a dedicated subnet with NSG rules. For private repos, use GitHub’s private networking (IPSEC tunnel or Azure Private Link). Egress can be locked down to specific endpoints via Azure Firewall or NAT Gateway with allow-lists for registries, package managers, and GitHub API endpoints only.
Q: How do we handle image vulnerabilities?
A: Three layers of defense: (1) Scan base images in ACR with Microsoft Defender for Containers – blocks deployment of images with critical CVEs. (2) Run Trivy or Snyk in the image build pipeline before pushing to ACR. (3) Use AKS image cleaner to remove stale images from nodes. Pin base images to digest (not tag) to prevent supply-chain attacks via tag mutation.
Q: What audit trail exists for runner activity?
A: Three sources: (1) GitHub Audit Log – captures who triggered workflows, which runners were used, and job outcomes (API + streaming to SIEM). (2) AKS diagnostic logs – pod lifecycle, resource consumption, network flows via Azure Monitor. (3) Application-level logs from the runner process itself, forwarded to Log Analytics. All three can be correlated by workflow run ID for end-to-end traceability.

🔀 Q&A – Migration & Vendor Lock-in

Addressing the #1 blocker to adoption

Migration Questions

Q: How long from kickoff to production?
A: Typical timeline: Week 1 – AKS cluster + ARC setup. Week 2 – First workflow migrated and validated. Week 3–4 – Cache optimization + monitoring. Week 5+ – Incremental migration of remaining workflows. Most teams have their first workflow running on self-hosted runners within 5 business days. Full fleet migration depends on workflow count, but 80% of value comes from the first 20% of workflows migrated.
Q: Can we migrate incrementally (not big-bang)?
A: Absolutely – this is the recommended approach. Use runner labels to route specific workflows to self-hosted runners while others continue on ubuntu-latest. Start with your slowest, most expensive workflow. Prove value, then expand. A single repository can have workflows running on both hosted and self-hosted runners simultaneously. No coordination required.
Q: What’s the rollback plan if something goes wrong?
A: Change runs-on: self-hosted back to runs-on: ubuntu-latest. One line. One commit. One minute. The workflow immediately runs on GitHub-hosted runners again. Your AKS infrastructure stays intact for debugging. This is the single strongest argument against lock-in fear – rollback is literally one YAML line.
Q: What team/skills do we need?
A: You need: (1) A platform/DevOps engineer comfortable with Kubernetes basics (deployments, PVCs, Helm). (2) A GitHub Actions author who understands workflow YAML. (3) Optional: a networking engineer for VNet peering / private link setup. Most teams don’t need a dedicated Kubernetes expert – ARC abstracts most of the complexity. If you can run helm install, you can run ARC.
Q: Do we have to rewrite our workflows?
A: No. The only required change is the runs-on: label. Everything else – steps, actions, secrets, environment variables – works identically. Optional enhancements (cache mounts, S3 sync) are additive. Your existing workflows are 100% compatible. We’ve migrated teams with 200+ workflows; the YAML diff is typically 1–3 lines per file.

Lock-in Questions

Q: Are we locked into Azure by choosing AKS?
A: ARC runs on any Kubernetes cluster – AKS, EKS, GKE, on-prem, even k3s. We recommend AKS for Azure-native workloads, but the runner controller is cloud-agnostic. You could run ARC on EKS tomorrow with the same Helm chart. Your workflows don’t reference Azure at all – they only reference runner labels.
Q: What if ARC pricing changes?
A: ARC (Actions Runner Controller) is open-source under Apache 2.0. GitHub doesn’t charge for ARC itself – you pay only for the AKS compute. Even if GitHub changed the runner registration model, the open-source controller could be forked. Your real cost is AKS nodes, which are standard Azure VMs with published pricing and reserved instance options.
Q: Why not just use AWS CodeBuild?
A: CodeBuild is solid for AWS-only shops. But: (1) No persistent cache across builds without custom S3 logic. (2) No native GitHub integration – you need CodePipeline or webhooks. (3) Vendor lock-in to AWS build specs. (4) Limited runner customization. If you’re all-in on AWS and don’t use GitHub Actions, CodeBuild is fine. If GitHub is your SCM, self-hosted runners give you tighter integration and more control.
Q: Why not Jenkins on Kubernetes?
A: Jenkins on K8s works and is battle-tested. Trade-offs: (1) You maintain the Jenkins control plane (upgrades, plugins, security patches). (2) Jenkinsfile syntax is separate from GitHub – two systems to learn. (3) No native GitHub status checks, PR integration, or Actions marketplace. (4) Jenkins controller is a single point of failure unless you set up HA. If you already have Jenkins expertise, it’s a valid choice. But for GitHub-native teams, ARC is simpler to operate.
Q: Is our cached data portable if we leave?
A: Yes. Caches are stored on standard PVCs (Azure Disk or Azure Files) or in S3/Blob Storage. They’re just files – node_modules, Maven .m2, Docker layers. No proprietary format. You can mount the same PVC on any Kubernetes cluster or copy the data anywhere. The cache strategy is yours, not GitHub’s.
Be HONEST about competition. CodeBuild has zero egress in-region — acknowledge it. Jenkins is fully open source — respect that. Our edge: GitHub native + persistent cache + operational simplicity. Never trash competitors — it undermines your credibility.
