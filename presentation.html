[TIME 11:30-13:30 | 2 min] Let me draw a clear distinction because people often confuse these two. Skills are TASK-SPECIFIC capabilities - they teach Copilot how to do something specialized, like deploy to your specific infrastructure or run your test framework. They're portable across VS Code, the CLI, and the coding agent. They can include scripts, examples, and documentation alongside the instructions. And they follow an open standard at agentskills.io. Custom Instructions, on the other hand, are ALWAYS-ON guidelines - your coding standards, naming conventions, architectural patterns. They're loaded into every conversation automatically. They work with glob patterns so you can have different instructions for different parts of your codebase. Think of it this way: Skills are like job training - specialized knowledge loaded on demand. Instructions are like company policy - always in effect.

[TIME 13:30-15:00 | 1.5 min] Creating a skill is straightforward - it's just a directory with a SKILL.md file as the entry point. The SKILL.md uses YAML frontmatter for metadata (name and description) followed by the actual instructions in Markdown. You can optionally add supporting resources: shell scripts for validation, code examples, reference docs. Personal skills go in ~/.copilot/skills/ and are available everywhere. Project skills go in .github/skills/ and are shared with the team. The key insight is that skills are directory-based, not single files. This means you can package complex workflows with supporting scripts and examples - the AI can access these resources when it needs them.

How Copilot Uses Skills

Three progressive levels of engagement

Level 1

🔍 Skill Discovery

Copilot reads name and description from YAML frontmatter. Lightweight metadata helps decide relevance to your prompt.

Level 2

📖 Instructions Loading

When request matches skill description, loads the SKILL.md body into context. Detailed instructions become available.

Level 3

📂 Resource Access

Access additional files — scripts, examples, documentation — only as needed. Maximum context efficiency.

✨ Skills are automatically activated based on your prompt — no manual selection needed.
[TIME 15:00-16:30 | 1.5 min] This is important to understand because it explains WHY skills are efficient. Three progressive levels. Level 1: Discovery - Copilot only reads the name and description from YAML frontmatter. Lightweight metadata matching. Level 2: Instructions Loading - if your prompt matches a skill's description, the full SKILL.md body gets loaded into context. Now Copilot has the detailed how-to. Level 3: Resource Access - only when the skill's instructions reference additional files does Copilot actually read those scripts, examples, or docs. This progressive loading is critical because context windows are limited. Skills don't waste tokens unless they're relevant. And the magic is automatic activation - just describe your task naturally and Copilot figures out which skills to load.

Prompt File Techniques

Key principles for effective prompt files

📎

Reference Other Files

Link to the files and resources the task should use as context

🎯

Single, Clear Purpose

Each prompt file should do one thing well — avoid overloading

📋

Describe WHAT and HOW

Include both the objective and the approach to achieve it

🔗

Modular System

Connect to your broader workflow — prompt files compose together

[TIME 18:00-19:30 | 1.5 min] Four key principles for effective prompt files. First, reference other files - use the #file syntax to pull in relevant source files as context. This grounds the AI in your actual code. Second, single clear purpose - don't try to make one prompt file do everything. 'Generate unit tests for a service' is good. 'Generate tests, update docs, and refactor' is too much. Third, describe WHAT and HOW - don't just say 'add error handling.' Say 'add error handling using our ErrorBoundary pattern, logging to our centralized logger, returning RFC 7807 problem details.' Fourth, keep them modular - prompt files compose together. A 'create-endpoint' prompt might reference a 'validation-pattern' prompt. This builds a reusable library your whole team can leverage.

[TIME 26:00-27:00 | 1 min] MCP, the Model Context Protocol, is one of the most important concepts to understand. Think of it like USB-C for AI. Just as USB-C gives you a universal port that works with any device, MCP gives AI applications a universal way to connect to external tools and data sources. Before MCP, every AI tool had its own proprietary way of connecting to databases, APIs, and file systems. MCP standardizes this so you build one integration and it works with Copilot, Claude, and any MCP-compatible client.

Core MCP Concepts Part 1

The three primary primitives

📊

Resources

Servers expose data and content to the AI model. Files, databases, API responses — all accessible as resources.

💬

Prompts

Servers define reusable prompt templates and workflows. Standardized interactions for common tasks.

🔧

Tools

LLMs interact with external systems, perform computations, and take actions in the real world.

[TIME 27:00-29:00 | 2 min] MCP has three primary primitives. Resources are how servers expose DATA to the AI - a read path. Files from your file system, rows from a database, responses from an API. The AI queries these to get context. Prompts are server-defined TEMPLATES - reusable prompt patterns for common tasks. A database MCP server might expose an 'analyze-query-performance' prompt template that the AI fills in with specific parameters. Tools are the ACTION primitives - how the AI interacts with external systems. Execute a database query, deploy to a server, create a Jira ticket, or call any API. Tools are the most commonly used MCP primitive because they let the AI DO things, not just read things.

Core MCP Concepts Part 2

Supporting infrastructure

🔄

Sampling

Servers request LLM completions through the client. Enables server-initiated AI interactions.

🌳

Roots

Define boundaries where servers can operate. Scoped access for security and context control.

🔀

Transports

The communication foundation between clients and servers. stdio, HTTP, and more.

[TIME 29:00-30:30 | 1.5 min] Three supporting concepts round out MCP. Sampling lets the MCP server request LLM completions THROUGH the client. Powerful but less commonly used - enables server-initiated AI interactions. Roots define BOUNDARIES where an MCP server can operate. A security feature - scope a file system server to only access certain directories. Transports are the communication layer - how clients and servers talk. Most common: stdio for local servers and HTTP/SSE for remote ones. For most of you, the key takeaway: Tools are 90% of what you'll use. Resources are useful for giving AI context. Roots keep everything secure.

[TIME 48:00-49:30 | 1.5 min] Think of single-session as a sprint and multi-session as a relay race. Sequential Orchestration

Ordered pipelines where outputs feed inputs

Pattern 1

🔗 Linear Pipeline

Scout → Scribe → Builder — strict ordered execution. Each agent specializes in one phase.

Pattern 2

📊 Dependency-Driven Order

Each agent's output feeds the next. Data flows through the pipeline with clear input/output contracts.

Pattern 3

🛡️ Quality Gates

Ensure flawed outputs don't cascade. Validation checkpoints between each stage prevent error propagation.

[TIME 49:30-51:00 | 1.5 min] Sequential orchestration is your default when outputs feed inputs - like an assembly line. Pattern 1, Linear Pipeline: Scout, Scribe, Builder. One agent explores and gathers info, the next writes a plan, the third implements. Each specializes in one phase. Pattern 2, Dependency-Driven Order: each agent's output feeds the next. Clear input/output contracts keep the chain reliable. Pattern 3, Quality Gates: validation checkpoints between stages preventing bad outputs from cascading. If the Scout's analysis is wrong, you don't want the Builder implementing a flawed plan. This is the safest pattern - use when correctness matters more than speed.

[TIME 59:00-60:30 | 1.5 min] Hooks are the ONLY mechanism for deterministic, non-AI-mediated control over the agent. This is critical - everything else (skills, instructions, agents) works through the AI, which means it's probabilistic. Hooks are imperative code - shell scripts or JavaScript - running OUTSIDE the AI loop at specific lifecycle events. Use preToolUse to intercept and validate tool calls before execution. Block dangerous commands, enforce security policies, validate inputs with real code, not AI judgment. Hooks fire during subagent execution and can be bundled inside plugins. This is the enterprise governance layer - if your security team needs to ensure certain commands never run, hooks are the answer.

Hooks Lifecycle

When hooks fire during agent execution

PreToolUse — Before any tool call
PostToolUse — After tool execution
Notification — On agent notifications
Stop — When agent stops/completes
SubagentStop — When subagent finishes
Hooks return exit codes: 0 = allow, non-0 = block the action
[TIME 60:30-61:30 | 1 min] Five lifecycle events where hooks fire. PreToolUse - before any tool call, inspect and block. PostToolUse - after execution, for logging or follow-up. Notification - when agent sends notifications. Stop - when agent stops or completes. SubagentStop - when a subagent finishes. Exit codes determine behavior: 0 means allow, non-zero means block. For preToolUse, this is your gating mechanism - return non-zero to prevent a tool call from executing. Simple but powerful.
